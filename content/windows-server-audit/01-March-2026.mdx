### CRITICAL-1: Windows Firewall Completely Disabled

**Finding:** The Windows Firewall is **OFF** on all three network profiles — Domain, Private, and Public. This means **zero network-level protection** exists on this server.

```
Domain Profile:  OFF
Private Profile: OFF
Public Profile:  OFF
```

**Risk:** Any machine on the network (or internet, if this server is reachable externally) can directly probe all open ports — RDP (3389), SMB (445), SQL (1433/1434), and any other service running. Combined with 200+ user accounts, brute-force or lateral movement attacks face no barrier.

**How to Fix:**

```powershell
# Step 1 — Enable firewall on all profiles
Set-NetFirewallProfile -Profile Domain,Private,Public -Enabled True

# Step 2 — Set default inbound policy to Block
Set-NetFirewallProfile -Profile Domain,Private,Public -DefaultInboundAction Block -DefaultOutboundAction Allow

# Step 3 — Allow only what is needed for a Domain Controller
# DNS
New-NetFirewallRule -DisplayName "Allow DNS Inbound" -Direction Inbound -Protocol TCP -LocalPort 53 -Action Allow
New-NetFirewallRule -DisplayName "Allow DNS UDP Inbound" -Direction Inbound -Protocol UDP -LocalPort 53 -Action Allow

# Kerberos
New-NetFirewallRule -DisplayName "Allow Kerberos" -Direction Inbound -Protocol TCP -LocalPort 88 -Action Allow
New-NetFirewallRule -DisplayName "Allow Kerberos UDP" -Direction Inbound -Protocol UDP -LocalPort 88 -Action Allow

# LDAP
New-NetFirewallRule -DisplayName "Allow LDAP" -Direction Inbound -Protocol TCP -LocalPort 389 -Action Allow
New-NetFirewallRule -DisplayName "Allow LDAPS" -Direction Inbound -Protocol TCP -LocalPort 636 -Action Allow

# SMB — restrict to internal subnet only (replace 10.x.x.0/24 with your subnet)
New-NetFirewallRule -DisplayName "Allow SMB Internal" -Direction Inbound -Protocol TCP -LocalPort 445 -RemoteAddress 10.0.0.0/8 -Action Allow

# RDP — restrict to IT admin IPs only
New-NetFirewallRule -DisplayName "Allow RDP IT Only" -Direction Inbound -Protocol TCP -LocalPort 3389 -RemoteAddress <IT_ADMIN_IP> -Action Allow

# Block all other inbound by default (already done above)
```

### CRITICAL-2: System Severely Under-Patched

**Finding:** Only **4 hotfixes installed**, with the last system update from **January 2024** — over **15 months ago**. Windows Server 2022 has had hundreds of security updates since then, including patches for critical vulnerabilities (PrintNightmare follow-ons, Kerberoasting mitigations, NTLM relay fixes, LDAP channel binding).

```
KB5033914  — Jan 2024
KB5012170  — Jan 2023
KB5034129  — Jan 2024
KB5034286  — Jan 2024
```

The **Windows Update service is currently in "Start Pending" state** — it is not running properly.

**Risk:** Unpatched Domain Controllers are the primary target in ransomware campaigns. An attacker with any foothold can escalate to Domain Admin using known, patched exploits (e.g., ZeroLogon CVE-2020-1472 if not patched, MS-RPRN attacks, etc.).

**How to Fix:**

```powershell
# Step 1 — Fix Windows Update service
Stop-Service wuauserv -Force
Start-Service wuauserv

# Step 2 — Install NuGet provider and PSWindowsUpdate module
Install-PackageProvider -Name NuGet -MinimumVersion 2.8.5.201 -Force
Install-Module -Name PSWindowsUpdate -Force

# Step 3 — Install all available updates
Import-Module PSWindowsUpdate
Get-WindowsUpdate -Install -AcceptAll -AutoReboot

# Step 4 — After reboot, configure automatic updates via Group Policy
# Computer Configuration > Administrative Templates > Windows Components > Windows Update
# Set "Configure Automatic Updates" = Enabled, option 4 (Auto download and schedule install)
# Schedule: Every Sunday at 2:00 AM (outside business hours)
```

**Schedule:** Patch every month on **Patch Tuesday +7 days** (third Tuesday of each month) to allow Microsoft to pull any bad patches first.

### CRITICAL-3: UAC Silently Elevates Without Prompting

**Finding:** User Account Control is configured to **elevate administrator operations silently**, without any security prompt.

```
EnableLUA:                    1   (UAC is on — good)
ConsentPromptBehaviorAdmin:   0   (Elevate without prompting — BAD)
PromptOnSecureDesktop:        0   (No secure desktop prompt — BAD)
```

**Risk:** Any malware or script running in an admin-level session can silently gain SYSTEM privileges with no user interaction or warning. This defeats the purpose of UAC entirely.

**How to Fix:**

```powershell
# Set UAC to prompt on secure desktop (recommended for servers)
Set-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System" `
    -Name "ConsentPromptBehaviorAdmin" -Value 2   # Prompt for credentials on secure desktop

Set-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System" `
    -Name "PromptOnSecureDesktop" -Value 1         # Use secure desktop
```

Or via Group Policy:
```
Computer Configuration > Windows Settings > Security Settings > Local Policies > Security Options
  - "User Account Control: Behavior of the elevation prompt for administrators" 
    = "Prompt for credentials on the secure desktop"
  - "User Account Control: Switch to the secure desktop when prompting for elevation"
    = Enabled
```

### CRITICAL-4: RDP Enabled with No Firewall

**Finding:** Remote Desktop Protocol is **fully enabled** (`fDenyTSConnections = 0`), and the firewall is completely off. RDP port 3389 is accessible to anyone who can reach this server.

**Risk:** RDP is one of the most targeted protocols globally. With no firewall and 200+ accounts, brute-force or credential-stuffing attacks are trivially easy. Ransomware operators actively scan for exposed RDP.

**How to Fix:**

```powershell
# Option A (Recommended): Restrict RDP to specific IPs via firewall (see CRITICAL-1)
# AND enable Network Level Authentication
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp" `
    -Name "UserAuthentication" -Value 1  # Enable NLA

# Option B: Disable RDP entirely if remote management is done via other means
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Terminal Server" `
    -Name "fDenyTSConnections" -Value 1

# Option C: Use a VPN gateway — require VPN connection before RDP is accessible
# (Highly recommended for an auditing firm)
```

### HIGH-2: Uncontrolled User Account Sprawl (200+ Accounts)

**Finding:** There are **200+ user accounts** in Active Directory with no evidence of a lifecycle management process. Numerous accounts have **never logged in** or have been **inactive for 180+ days** and remain enabled:

**Accounts that have NEVER logged in (enabled):**
`purayil`, `Vignesh`, `srinivas`, `neethu`, `kaveri`, `userea`, `salih`, `manish`, `raja.a`, `Siva`, `Krutika.M`, `padma`, `venkatraman`, `maryam`, `mcaaccounts`, `newproject`

**Accounts inactive 180+ days (sample):**
`athira`, `sathish`, `muthumeena`, `nahla`, `preetham`, `rekha`, `jeleshia`, `akshay`, `devang`, `rahul`, `pavan`, `sandhya`, `shubham`

**Risk:** Inactive accounts are a primary vector for persistence. An attacker can use a compromised inactive account that no one monitors for months.

**How to Fix:**

```powershell
# Step 1 — Find all accounts inactive for 90+ days
$cutoffDate = (Get-Date).AddDays(-90)
$staleUsers = Get-ADUser -Filter {
    LastLogonDate -lt $cutoffDate -and Enabled -eq $true
} -Properties LastLogonDate, Department, Manager |
Select-Object Name, SamAccountName, LastLogonDate, Department, Manager

$staleUsers | Export-Csv "C:\Audit\StaleAccounts_$(Get-Date -Format 'yyyyMMdd').csv" -NoTypeInformation

# Step 2 — Disable (do not delete) accounts inactive 90+ days
foreach ($user in $staleUsers) {
    Disable-ADAccount -Identity $user.SamAccountName
    Set-ADUser -Identity $user.SamAccountName -Description "Auto-disabled $(Get-Date -Format 'yyyy-MM-dd') - inactive 90 days"
}

# Step 3 — Move disabled accounts to a dedicated OU
New-ADOrganizationalUnit -Name "Disabled Users" -Path "DC=mcagulf,DC=local"
foreach ($user in $staleUsers) {
    Move-ADObject -Identity (Get-ADUser $user.SamAccountName).DistinguishedName `
        -TargetPath "OU=Disabled Users,DC=mcagulf,DC=local"
}

# Step 4 — Delete accounts disabled for 180+ days (after review)
# Run this monthly as a scheduled task
```

**Process to implement:**
- HR must notify IT **on the day of employee departure** 
- IT must disable the account **same day** and revoke access
- Accounts disabled for 180 days are deleted
- Quarterly access review of all active accounts

### HIGH-4: Administrator Account Password Unchanged Since 2022

**Finding:** The built-in `Administrator` account password has **not been changed since the server was installed on December 26, 2022** — over 3 years ago. The `PasswordExpires` field is blank, suggesting the password is set to never expire.

**Risk:** The built-in Administrator is the highest-value target on any Windows domain. A 3-year-old password dramatically increases the probability it has been compromised or that the original installer password is still in use.

**How to Fix:**

```powershell
# Step 1 — Immediately change the Administrator password
$newPassword = Read-Host "Enter new Administrator password" -AsSecureString
Set-ADAccountPassword -Identity Administrator -NewPassword $newPassword -Reset

# Step 2 — Rename the built-in Administrator account to reduce targeting
Rename-LocalUser -Name "Administrator" -NewName "mcagulf_sysadmin"

# Step 3 — Create a decoy account named "Administrator" with no privileges
# (Honeypot - any login attempt on it triggers an alert)
New-LocalUser -Name "Administrator" -NoPassword -Description "DO NOT USE"
Disable-LocalUser -Name "Administrator"

# Step 4 — Store the real admin password in a PAM vault (see Improvements section)

# Step 5 — Enable LAPS (Local Administrator Password Solution) for all machines
# This auto-rotates local admin passwords across all domain computers
Install-Module -Name LAPS
# Or install via: https://www.microsoft.com/en-us/download/details.aspx?id=46899
```

### HIGH-5: Tally and SQL Server Running on Domain Controller

**Finding:** The following business applications are installed and auto-starting on the **Primary Domain Controller**:

```
SQL Server Express (MSSQL$SQLEXPRESS) — Auto, Stopped
SQL Server Agent (SQLAgent$SQLEXPRESS) — Auto, Stopped  
Tally Gateway Server 11.0 — Auto, Stopped
```

**Risk:** Domain Controllers should run **only Active Directory services**. Running application software on a DC means:
1. A vulnerability in Tally or SQL can be escalated to full DC compromise
2. SQL injection or application-level attack → Domain Admin
3. Violates Microsoft best practices and most compliance frameworks (ISO 27001, SOC 2)

**How to Fix:**
1. Provision a **separate dedicated application server** (physical or VM)
2. Migrate SQL Server Express and Tally Gateway to that server
3. Remove SQL Server and Tally from the DC
4. If budget is constrained, at minimum: run these as **low-privilege service accounts** with no domain admin rights

### HIGH-6: WSL (Windows Subsystem for Linux) on Domain Controller

**Finding:** Windows Subsystem for Linux (`WSLService`) is installed and set to auto-start on the Domain Controller.

**Risk:** WSL creates a Linux subsystem that can bypass Windows security controls, Windows Defender scanning, and audit logging. Malware or insiders can use WSL to run Linux-native tools (including attack tools) that are invisible to Windows security monitoring. WSL has no place on a Domain Controller.

**How to Fix:**

```powershell
# Disable WSL completely
Disable-WindowsOptionalFeature -Online -FeatureName Microsoft-Windows-Subsystem-Linux -NoRestart
Disable-WindowsOptionalFeature -Online -FeatureName VirtualMachinePlatform -NoRestart

# Set the service to disabled
Set-Service -Name WSLService -StartupType Disabled
Stop-Service -Name WSLService -Force

# Reboot required
Restart-Computer -Force
```

**Risk:** Remote registry access allows attackers or malicious insiders to modify security settings, install persistence mechanisms, or extract credential material from the registry remotely without needing a local session.

### MEDIUM-2: Event Logs Set to Overwrite (Audit Trail at Risk)

**Finding:** All event logs are configured to **overwrite old events as needed**, with small log sizes:

```
Application Log:  20 MB  — OverwriteAsNeeded
Security Log:    128 MB  — OverwriteAsNeeded
System Log:       20 MB  — OverwriteAsNeeded
```

**Risk:** On a busy Domain Controller, the Security log (which captures all authentication events) can fill up in days. Once full, oldest events are lost, eliminating the ability to investigate a breach that occurred weeks ago. This is a **compliance failure** for an auditing firm.

**How to Fix:**

```powershell
# Increase log sizes
Limit-EventLog -LogName Security     -MaximumSize 1GB
Limit-EventLog -LogName Application  -MaximumSize 256MB
Limit-EventLog -LogName System       -MaximumSize 256MB

# Change retention policy — do NOT overwrite, archive instead
# This requires setting up log archival (see SIEM recommendation in Part 3)

# Or configure via Group Policy:
# Computer Configuration > Administrative Templates > Windows Components > Event Log Service
#   Security: Maximum Log Size = 1048576 KB (1 GB)
#   Security: Retain old events = Enabled (requires archival solution)

# Configure Advanced Audit Policy (Critical for compliance)
auditpol /set /subcategory:"Logon" /success:enable /failure:enable
auditpol /set /subcategory:"Account Logon" /success:enable /failure:enable
auditpol /set /subcategory:"Account Management" /success:enable /failure:enable
auditpol /set /subcategory:"DS Access" /success:enable /failure:enable
auditpol /set /subcategory:"Object Access" /success:enable /failure:enable
auditpol /set /subcategory:"Policy Change" /success:enable /failure:enable
auditpol /set /subcategory:"Privilege Use" /success:enable /failure:enable
auditpol /set /subcategory:"Process Creation" /success:enable /failure:enable
```

### MEDIUM-3: NTP Not Syncing to External Time Source

**Finding:** The server is using its own internal clock (`LOCL` — Free-running System Clock) instead of an authoritative external NTP source.

```
Source: Free-running System Clock
ReferenceId: LOCL
```

**Risk:** Log timestamps will drift over time. In a security investigation or compliance audit, inconsistent timestamps across systems (server vs. firewall vs. endpoints) make it impossible to accurately reconstruct an incident timeline.

**How to Fix:**

```powershell
# Configure NTP to sync from authoritative external sources
w32tm /config /manualpeerlist:"time.windows.com pool.ntp.org" /syncfromflags:manual /reliable:yes /update

# Restart Windows Time service
Restart-Service w32tm

# Force sync and verify
w32tm /resync
w32tm /query /status
```

### 3.5 Local Administrator Password Solution (LAPS)

Every domain-joined computer should have a **unique, auto-rotating local administrator password**. Currently, if the local admin password is the same across all machines, compromising one machine gives admin access to all.

```powershell
# Install LAPS on the DC
Add-WindowsFeature RSAT-AD-PowerShell
# Download LAPS.x64.msi from Microsoft
# Run installer and select "Management Tools" + "FAT client extension"

# Extend AD schema
Update-AdmPwdADSchema

# Set permissions — which groups can READ the password
Set-AdmPwdComputerSelfPermission -OrgUnit "OU=Workstations,DC=mcagulf,DC=local"
Set-AdmPwdReadPasswordPermission -OrgUnit "OU=Workstations,DC=mcagulf,DC=local" -AllowedPrincipals "Domain Admins"

# Deploy LAPS client via Group Policy (MSI deployment)
# Passwords auto-rotate every 30 days and are stored in AD
```

### 3.6 File Share Access Auditing & DLP

Given that client audit data, HR records, and financial data are on these shares, you need **object-level access auditing**:

```powershell
# Enable auditing on sensitive shares
$acl = Get-Acl "C:\Data Directory\Audit"
$auditRule = New-Object System.Security.AccessControl.FileSystemAuditRule(
    "Everyone", "ReadData,WriteData,Delete", 
    "ContainerInherit,ObjectInherit", "None", "Success,Failure"
)
$acl.SetAuditRule($auditRule)
Set-Acl "C:\Data Directory\Audit" $acl

# Repeat for: HR, Partners, Financial Services, Corporate Finance, GRC, etc.
```

Also consider **Microsoft Purview Information Protection** (formerly AIP):
- Classify documents as Confidential / Client Confidential / Internal
- Encrypt files so they are unreadable outside the organization
- Prevent printing/forwarding/copy-paste of sensitive documents

### 3.8 Backup & Recovery (Ransomware Resilience)

Verify that backups exist and test them. Key requirements:

- **3-2-1-1-0 Rule:** 3 copies, 2 different media, 1 offsite, 1 offline/airgapped, 0 errors on restore test
- Backups must be **immutable** — ransomware operators now target backup systems first
- **Recovery Time Objective (RTO):** How long can the firm operate without access to client data?
- **Recovery Point Objective (RPO):** Maximum acceptable data loss (hourly? daily?)

```powershell
# Check if Windows Server Backup is installed and configured
Get-WindowsFeature Windows-Server-Backup
Get-WBPolicy  # View current backup policy

# Recommended: Azure Backup for off-site immutable backups
# Install Azure Backup agent and configure:
#   - Daily backup at 11 PM
#   - Retention: 30 daily, 12 monthly, 7 yearly
#   - Enable soft-delete (14-day protection against ransomware deletion)
```

## Part 5 — Compliance Considerations

As an **auditing firm storing client financial data**, you are likely subject to:

| Framework | Requirement | Current Gap |
|-----------|------------|-------------|
| **GDPR / UAE PDPL** | Access controls, audit trails, breach notification | Gaps in access control, logging |
| **ISO 27001** | Information security management system | No formal ISMS documented |
| **SOC 2 Type II** | Security, availability, confidentiality controls | Multiple control failures |
| **ISAE 3402** | Internal controls over financial reporting | Shared accounts break audit trail |
| **UAE Cybersecurity Law** | Critical infrastructure protections | Firewall off, patching gaps |

**Key compliance action:** The use of **shared and generic accounts** (`common`, `audit1`, `policy`) is a direct violation of audit trail requirements under all these frameworks. Individual accountability is required — every action must be traceable to a specific named person.
